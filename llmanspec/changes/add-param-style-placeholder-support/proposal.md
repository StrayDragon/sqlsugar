## Why

当前 Jinja2 可视化编辑器仅支持 Jinja2 模板变量（`{{ var }}`）和 SQLAlchemy 的 `:param` 占位符（通过 R-J2E-006）。然而，在实际 SQL 开发中，不同的数据库驱动和 ORM 使用不同的参数占位符风格（如 `?`、`:1`、`%s`、`%(name)s`、`$1` 等）。这限制了编辑器的通用性，用户无法在一个统一的界面中处理多种参数风格的 SQL 模板。

此外，现有的单遍解析架构无法支持多种分析器的组合使用，需要升级为多遍遍历架构以支持灵活的分析器组合。

## What Changes

- **多遍遍历架构**：实现可插拔的分析器管道（Analyzer Pipeline），支持按优先级依次执行多个分析器
- **参数风格分析器**：实现以下参数风格的提取器：
  - `named`: `:param`（已部分支持，需完善）
  - `numeric`: `:1`, `:2`, ...
  - `pyformat`: `%(param)s`
  - `asyncpg`: `$1`, `$2`, ...
- **分析器选择 UI**：在编辑器工具栏中添加分析器选择组件，支持：
  - 自动检测模式（默认）
  - 手动多选勾选特定分析器
  - 分析器优先级排序
- **参数可视化增强**：在变量编辑器中统一展示 Jinja2 变量和参数占位符，提供一致的编辑体验
- **配置扩展**：新增 `sqlsugar.paramStyle.enabledAnalyzers` 配置项，允许用户预设默认启用的分析器

## Capabilities

1. **多分析器管道**：系统 MUST 支持按优先级依次执行多个分析器，每个分析器负责提取特定类型的参数
2. **参数风格提取**：系统 MUST 支持提取 named、numeric、pyformat、asyncpg 风格的参数占位符
3. **分析器选择 UI**：系统 MUST 在编辑器工具栏中提供分析器选择组件，支持自动检测和手动多选
4. **参数统一展示**：系统 MUST 在变量编辑器中统一展示 Jinja2 变量和参数占位符
5. **配置支持**：系统 MUST 提供配置项让用户预设默认启用的分析器

## Impact

- **向后兼容**：现有 Jinja2 模板处理功能不受影响，默认启用 Jinja2 分析器
- **性能影响**：多遍遍历会略微增加解析时间，但通过优先级和短路机制可控制在可接受范围内
- **UI 变更**：工具栏将新增分析器选择组件，需要相应的 UI 空间
- **测试覆盖**：需要为每个参数风格分析器编写单元测试，确保提取准确性
